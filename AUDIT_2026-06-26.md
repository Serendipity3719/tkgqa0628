## 3. 当前 Agent 是否属于 Deep Agent?

**判断: 否。它是一个「单工具 ReAct 式 tool-use agent」, 不是强意义的 Deep Agent。**
逐项依据代码:

| 能力 | 有/无 | 代码证据 |
|---|---|---|
| **Planning (规划)** | **无** | 不存在任何「生成计划 / 分解子目标」的步骤。所谓「工作流」是写死在 SYSTEM 提示里的静态 SOP (`agent_nav.py:42-51`), 由人预设, **非模型自主规划**, 也不被存储/追踪。 |
| **Multi-step reasoning (多步推理)** | **有** | `agent_solve` 是 ReAct 循环 `:97-120`, 每轮「推理→发命令→看结果」, 最多 15 轮。这是真实的多步交互。 |
| **Tool calling (工具调用)** | **有, 但受限** | 唯一工具 = 只读 bash (`run_bash :60-78`); 调用方式是**文本解析 ```bash 代码块** (`_BLOCK :55`, 解析 `:106-109`), **非原生 function-calling API**。 |
| **Memory / State (记忆/状态)** | **弱, 近乎无持久状态** | 仅靠 `messages` 列表累积上下文 (`:94, :99, :114`) = 单回合上下文窗口。**无外部状态存储, 无可靠跨命令 scratchpad** (serve 端每条命令是全新 `bash -lc`, shell 变量不跨轮存活)。v2 schema 设计的 `STATE:` 账本**当前代码并未实现**; `exp_trace` 里的 `reconstructed_state` 是**事后推断**, 非 agent 自维护。 |
| **Self-reflection / Backtracking (反思/回溯)** | **仅提示层, 实际几乎不触发** | 回溯只写在提示 (`:46` 「空结果就回溯」) 和 skill 文件里, **无显式反思/校验/自我批评模块**。是否回溯全靠模型自己注意到 `[空结果]` observation。实测 `exp_eval` 显示 `avg_backtracks≈0.07` (correct 0.02) —— **回溯名义存在、实际极少发生**。 |

**结论**: 具备 [多步推理 + 工具调用] 两项; 缺失 [规划 + 持久状态/记忆 + 真实反思回溯] 三项。
因此它处于 **ReAct agent** 层级, 尚未达到「Deep Agent」(需 planner + 持久 memory + reflection)。
这与论文「Navigate」叙事一致 (论文强调导航而非复杂规划), 但**不要在论文里把它描述为 Deep Agent**。

## 5. 当前实验目标重新判断

**判断: 当前代码主要验证的是 B —— Skill Navigation Policy (在路由正确的前提下)。
它尚未验证 A, 只在「oracle 路由」意义下近似验证 C。**

依据:

1. **A (Skill Routing) 当前被旁路。** `agent_nav.py:92` 把 `qtype` 直接喂给 Agent;
   Router 退化为「按 qtype 查 SKILLS.md 表 → cat 对应 skill」, 几乎无难度。
   `exp_eval` 实测 routing≈97%、`routing_loss≈3%`, 正是因为这是 **oracle 上界, 非真实路由**。

2. **真正被考验、被度量的是 B (Navigation Policy)。** 系统的难点全在导航段:
   实体绑定 (catalog) / 关系映射 (_relations) / 方向判定 / 时间位置选择 / awk 取证。
   `exp_eval` 的 `navigation_loss | routing 正确 ≈ 14.4%` 量化的就是这一段, 是主要损耗源
   (15 个失败中 11–12 个是 `temporal_reasoning_error`)。

3. **C (整体 Skill-based KGQA) 只在「给定 qtype」下成立。** 端到端 85% 这个数字
   **包含了被旁路的路由阶段**, 所以它是「C 减去路由难度」。要主张完整的 C, 必须用
   `exp_runner.py --routing-mode blind` 让 Agent 仅凭问句自选 skill。

| 论文若要主张… | 当前代码是否支持 | 需要的动作 |
|---|---|---|
| B: 导航策略有效 | ✅ 已支持 (现有 oracle trace 即可分析) | 直接用 `exp_eval` |
| A: 路由有效 | ❌ 未支持 (qtype 泄露) | 跑 `--routing-mode blind` |
| C: 端到端 KGQA 有效 | ⚠ 仅 oracle 路由下成立 | 跑 blind, 报 blind 的 e2e |

> 一句话: **现状是「在已知问题类型的前提下, 验证文件系统导航能否取出正确答案」=
> Skill Navigation Policy 的验证。** 路由的真实有效性 (A) 与无泄露的端到端 (C) 还是空白,
> 而这恰好是已搭好的 `--routing-mode blind` ablation 要补的一刀。
